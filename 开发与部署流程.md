## 二、环境运行逻辑

### 2.1 开发环境

| 服务 | 技术栈 | 代码同步 | 热更新 | 访问地址 |
|------|--------|----------|--------|----------|
| 后端 | golang:1.24-alpine | backend → /app | 需重启容器 | http://localhost:8080 |
| 前端(管理端) | node:20-alpine | frontend → /app | Vite自动热更新 | http://localhost:3000 |
| C端 | node:20-alpine | c端 → /app | Vite自动热更新 | http://localhost:3001 |
| MySQL | mysql:8.0 | 持久化到 docker/data/mysql | - | 3306 |
| Redis | redis:7.0 | 持久化到 docker/data/redis | - | 6379 |
| RabbitMQ | rabbitmq:3.10-management | 持久化到 docker/data/rabbitmq | - | 5672/15672 |
| Nginx | nginx:1.20 | - | - | 80/443 |

### 2.2 生产环境

- **后端**：多阶段构建，使用预构建的可执行文件运行
- **前端/C端**：构建产物存储在 `dist` 目录，Nginx提供静态文件服务
- **Nginx**：作为反向代理，处理静态文件和API请求转发

## 三、API配置

### 3.1 前端API配置

**文件**：`frontend/src/api/auth.js`

```javascript
const API_BASE_URL = '/api';
// 或使用环境变量
// const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || '/api';
```

**Vite配置**（`frontend/vite.config.js`）：

```javascript
server: {
  port: 3000,
  proxy: {
    '/api': {
      target: import.meta.env.VITE_API_PROXY_TARGET || 'http://backend:8080',
      changeOrigin: true
    }
  }
}
```

**环境变量**（`frontend/.env`）：

```
VITE_API_BASE_URL=/api
VITE_API_PROXY_TARGET=http://backend:8080
```

### 3.2 C端API配置

**文件**：`c端/src/api/auth.js`

```javascript
const API_BASE_URL = '/api';
// 或使用环境变量
// const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || '/api';
```

**Vite配置**（`c端/vite.config.js`）：

```javascript
server: {
  port: 3001,
  proxy: {
    '/api': {
      target: import.meta.env.VITE_API_PROXY_TARGET || 'http://backend:8080',
      changeOrigin: true
    }
  }
}
```

### 3.3 Nginx配置

**文件**：`docker/nginx/nginx.conf`

```nginx
# 管理端
server {
    listen 80;
    server_name admin.example.com;
    root /usr/share/nginx/admin;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://backend:8080/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# C端商城
server {
    listen 80;
    server_name example.com;
    root /usr/share/nginx/c端;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://backend:8080/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## 四、开发流程

### 4.1 环境准备

1. **安装Docker**：确保本地安装了Docker和Docker Compose
2. **克隆代码**：从GitHub克隆项目代码
3. **配置环境变量**：
   - 后端：复制 `.env.example` 为 `.env` 并修改配置
   - 前端：无需特殊配置

### 4.2 启动开发环境

```bash
# 进入docker目录
cd docker

# 启动所有服务
docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看日志
docker-compose logs -f
```

### 4.3 开发工作流

#### 后端开发
1. 修改 `backend` 目录下的代码
2. 代码会自动同步到容器中
3. Go服务会自动重新运行
4. 通过 `http://localhost:8080` 访问API

#### 前端开发（管理端）
1. 修改 `frontend` 目录下的代码
2. 代码会自动同步到容器中
3. Vite会自动热更新
4. 通过 `http://localhost:3000` 访问前端

#### C端开发
1. 修改 `c端` 目录下的代码
2. 代码会自动同步到容器中
3. Vite会自动热更新
4. 通过 `http://localhost:3001` 访问C端

### 4.4 代码提交

1. **代码规范**：遵循Go和Vue的代码规范
2. **提交信息**：使用清晰的提交信息
3. **分支管理**：遵循GitFlow分支管理规范

## 五、部署流程

### 5.1 构建生产镜像

```bash
# 进入docker目录
cd docker

# 构建所有服务
docker-compose build
```

### 5.2 部署生产环境

#### 配置修改
- 修改 `docker-compose.yml` 中的环境变量
- 确保数据持久化目录存在
- 配置Nginx反向代理

#### 启动服务

```bash
# 启动生产服务（设置环境变量）
NODE_ENV=production RESTART_POLICY=always docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看日志
docker-compose logs -f
```

### 5.3 部署验证

1. **后端API**：访问 `http://服务器IP:8080/health` 验证健康状态
2. **前端应用（管理端）**：访问 `http://admin.服务器IP` 验证前端是否正常
3. **C端商城**：访问 `http://服务器IP` 验证C端是否正常
4. **数据库连接**：验证MySQL、Redis、RabbitMQ连接正常

### 5.4 监控与维护

1. **日志管理**：使用 `docker-compose logs` 查看服务日志
2. **服务监控**：配置健康检查和监控系统
3. **备份策略**：定期备份 `docker/data` 目录
4. **更新部署**：
   - 停止服务：`docker-compose down`
   - 拉取最新代码：`git pull`
   - 重新构建：`docker-compose build`
   - 启动服务：`docker-compose up -d`

## 六、API配置验证与技术优势

### 6.1 验证方法

#### 开发环境
1. 启动前端开发服务器：`npm run dev`
2. 启动后端服务：`docker compose up -d`
3. 打开浏览器访问 `http://localhost:3000`
4. 测试API请求：
   - 打开浏览器开发者工具
   - 检查网络请求，确认所有API请求都通过 `localhost:3000/api` 发送
   - 验证请求是否成功

#### 生产环境
1. 构建前端项目：`npm run build`
2. 启动Docker容器：`docker compose up -d`
3. 打开浏览器访问 `http://localhost`（Nginx默认端口）
4. 测试API请求：
   - 打开浏览器开发者工具
   - 检查网络请求，确认所有API请求都通过 `http://localhost/api` 发送
   - 验证请求是否成功

### 6.2 技术优势

1. **环境适配**：无论是开发环境还是生产环境，都使用相同的API路径
2. **跨域处理**：开发环境通过Vite代理处理跨域，生产环境通过Nginx处理
3. **配置集中**：所有API路径配置统一管理
4. **部署灵活**：后端服务地址变更时，只需修改代理配置
5. **代码一致性**：前端代码无需为不同环境修改API路径

## 七、常见问题与解决方案

### 7.1 服务启动失败
- **检查日志**：`docker-compose logs 服务名`
- **端口冲突**：确保端口未被占用
- **依赖问题**：检查数据库和中间件是否正常

### 7.2 代码同步问题
- **检查挂载卷**：确保本地目录正确挂载
- **权限问题**：检查文件权限
- **路径问题**：确保路径配置正确

### 7.3 性能优化
- **资源限制**：在 `docker-compose.yml` 中配置资源限制
- **缓存策略**：合理配置Redis缓存
- **数据库优化**：添加索引，优化查询

### 7.4 安全配置
- **环境变量**：敏感信息使用环境变量
- **网络隔离**：使用Docker网络隔离服务
- **访问控制**：配置Nginx访问控制
