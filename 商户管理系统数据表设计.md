# 商城后台多商户系统数据表设计

## 1. 现有系统分析

### 1.1 现有表结构
| 表名 | 用途 | 关键字段 |
|------|------|--------|
| `users` | 用户表 | `id`, `username`, `password`, `role_id`, `status` |
| `roles` | 角色表 | `id`, `name`, `description`, `status` |
| `permissions` | 权限表 | `id`, `name`, `code`, `description`, `status` |
| `role_permissions` | 角色权限关联表 | `role_id`, `permission_id` |
| `orders` | 订单表 | `id`, `order_no`, `user_id`, `merchant_id`, `total_amount`, `status` |
| `products` | 商品表 | `id`, `name`, `description`, `price`, `stock`, `category`, `merchant_id`, `status` |
| `activities` | 活动表 | `id`, `name`, `type`, `start_time`, `end_time`, `status`, `created_by` |
| `shipping` | 物流表 | `id`, `order_id`, `tracking_no`, `status` |

### 1.2 现有角色
- 超级管理员
- 平台管理员
- 商户账号

### 1.3 现有权限
- user:manage (用户管理)
- role:manage (角色管理)
- product:manage (商品管理)
- order:manage (订单管理)
- activity:manage (活动管理)
- shipping:manage (物流管理)

## 7. 表结构SQL语句

### 7.1 商户表

```sql
CREATE TABLE `merchants` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `name` varchar(100) COLLATE utf8mb4_unicode_ci NOT NULL,
  `contact_name` varchar(50) COLLATE utf8mb4_unicode_ci NOT NULL,
  `contact_phone` varchar(20) COLLATE utf8mb4_unicode_ci NOT NULL,
  `email` varchar(100) COLLATE utf8mb4_unicode_ci NOT NULL,
  `address` varchar(255) COLLATE utf8mb4_unicode_ci NOT NULL,
  `business_license` varchar(255) COLLATE utf8mb4_unicode_ci NOT NULL,
  `tax_number` varchar(50) COLLATE utf8mb4_unicode_ci NOT NULL,
  `audit_status` enum('pending','approved','rejected') COLLATE utf8mb4_unicode_ci DEFAULT 'pending',
  `status` enum('active','inactive') COLLATE utf8mb4_unicode_ci DEFAULT 'inactive',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `approved_at` datetime DEFAULT NULL,
  `approved_by` int(11) DEFAULT NULL,
  PRIMARY KEY (`id`),
  INDEX `idx_merchants_status` (`status`),
  INDEX `idx_merchants_audit_status` (`audit_status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 7.2 商户用户关联表

```sql
CREATE TABLE `merchant_users` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `merchant_id` int(11) NOT NULL,
  `user_id` int(11) NOT NULL,
  `role` varchar(20) COLLATE utf8mb4_unicode_ci NOT NULL,
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `unique_merchant_user` (`merchant_id`, `user_id`),
  INDEX `idx_merchant_users_merchant_id` (`merchant_id`),
  INDEX `idx_merchant_users_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 7.3 商户审核表

## 9. 设计说明

1. **商户表 (`merchants`)**：
   - 存储商户基本信息
   - 包含审核状态和商户状态
   - 审核通过后商户状态自动变为active

2. **商户用户关联表 (`merchant_users`)**：
   - 关联商户和用户
   - 支持一个商户多个用户
   - 定义商户内角色(owner, manager, staff)

3. **商户审核表 (`merchant_audit`)**：
   - 记录商户注册和信息更新的审核流程
   - 使用JSON类型存储审核前后的数据
   - 支持审核状态跟踪

4. **商户银行信息表 (`merchant_bank`)**：
   - 存储商户银行账户信息
   - 支持设置默认账户
   - 一个商户可以有多个银行账户

5. **商户提现记录表 (`merchant_withdraw`)**：
   - 记录商户提现申请和处理状态
   - 包含提现金额、银行信息等
   - 支持提现状态跟踪

6. **商户对账单表 (`merchant_statement`)**：
   - 生成商户定期对账单
   - 记录时间段内的交易汇总
   - 支持对账状态跟踪

7. **商品分类表 (`product_categories`)**：
   - 支持多级分类
   - 支持分类排序
   - 与商品表关联

8. **商品SKU表 (`product_skus`)**：
   - 存储商品的不同属性组合
   - 每个SKU有独立的价格和库存
   - 使用JSON存储SKU属性

9. **商品图片表 (`product_images`)**：
   - 存储商品的多张图片
   - 支持设置主图
   - 支持图片排序

10. **订单明细表 (`order_items`)**：
    - 存储订单中的商品明细
    - 记录购买时的商品信息和价格
    - 支持关联SKU信息

11. **支付记录表 (`payments`)**：
    - 记录订单支付信息
    - 支持多种支付方式
    - 跟踪支付状态

12. **地址表 (`addresses`)**：
    - 存储用户的收货地址
    - 支持设置默认地址
    - 与用户表关联

13. **活动规则表 (`activity_rules`)**：
    - 存储活动的具体规则
    - 使用JSON存储规则配置
    - 与活动表关联

14. **活动商品表 (`activity_products`)**：
    - 存储参与活动的商品
    - 记录活动价格和库存
    - 与活动表和商品表关联

15. **活动效果统计表 (`activity_stats`)**：
    - 统计活动的效果数据
    - 包含浏览、参与、订单等数据
    - 与活动表关联

16. **权限设计**：
    - 新增商户、商品、订单、活动相关权限
    - 合理分配给不同角色
    - 确保权限控制精确到操作级别

## 10. 注意事项

1. **数据一致性**：
   - 商户审核通过后，自动更新商户状态
   - 商户状态变更时，同步更新关联数据
   - 订单状态变更时，同步更新支付和物流状态
   - 商品库存变更时，同步更新SKU库存

2. **性能考虑**：
   - 为常用查询字段添加索引
   - 合理设计表结构，避免冗余字段
   - 对于JSON字段，考虑使用适当的索引策略
   - 活动统计数据可考虑使用缓存机制

3. **安全性**：
   - 敏感信息如银行账号需要加密存储
   - 提现操作需要多重验证
   - 支付信息需要加密传输
   - 商户审核流程需要严格的权限控制

4. **扩展性**：
   - 表结构设计预留扩展字段
   - 权限系统支持后续功能扩展
   - 商品属性和活动规则使用JSON格式，便于灵活配置
   - 支付方式设计支持后续扩展

5. **与现有系统集成**：
   - 保持与现有表结构的一致性
   - 确保订单表和商品表的merchant_id字段与商户表关联
   - 新表的字段命名和类型与现有表保持一致
   - 遵循现有系统的状态字段设计规范

6. **业务逻辑**：
   - 活动商品库存管理需要独立于普通商品库存
   - 订单创建时需要扣减相应的商品库存
   - 订单取消时需要恢复商品库存
   - 商户提现需要与对账单关联

7. **数据迁移**：
   - 现有系统的merchant_id字段需要与新商户表关联
   - 数据迁移过程中需要确保数据一致性
   - 迁移后需要验证数据完整性
