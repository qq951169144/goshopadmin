### 2.3 商品SKU管理功能

#### 2.3.1 商品创建逻辑调整

**问题分析**：
当前商品创建时，没有设置SKU的情况下，只有products表的价格和库存，导致库存管理逻辑不一致，违背了"products.stock = 所有SKU库存之和"的绝对规则。

**解决方案**：采用方案1 - 创建商品时自动创建默认SKU

**具体实现**：

1. **数据库调整**：
   - products表的price字段标记为废弃（保留字段但不使用）✅
   - 所有价格信息统一由product\_skus表管理✅
   - products.stock字段保持，但作为所有SKU库存的总和展示✅
2. **商品创建逻辑**：
   - 创建商品时，自动创建一个默认SKU记录✅
   - 默认SKU的attributes字段为空或设置为"默认"✅
   - 默认SKU的价格和库存使用商品创建时传入的数据✅
   - products.stock自动设置为默认SKU的库存值✅
3. **默认SKU特征**：
   - sku\_code：自动生成，如"PROD-{product\_id}-DEFAULT"✅
   - attributes：空JSON对象或{"type": "default"}✅
   - price：等于商品创建时的价格✅
   - stock：等于商品创建时的库存✅
   - status：'active'✅
4. **后续SKU管理**：
   - 可以添加更多SKU规格✅
   - 可以修改默认SKU的价格和库存✅
   - 删除最后一个SKU时需要特殊处理（禁止删除或重新创建默认SKU）✅
5. **前端调整**：
   - 商品创建表单中，价格和库存字段仍然显示✅
   - 创建成功后，自动显示默认SKU信息✅
   - 在SKU管理界面中，默认SKU可以编辑但不能删除✅

**优势**：

- 保持了SKU作为库存管理单位的统一性
- 逻辑简单清晰，所有库存都通过SKU管理
- 完美符合"products.stock = 所有SKU库存之和"的绝对规则
- 后续扩展性强，需要添加SKU时直接修改即可

### 2.4 库存管理关系分析

#### 分析目标

- 分析`products`、`product_skus`、`activity_products`三张表的库存管理关系
- 确定库存更新和同步机制
- 确保活动期间不会超卖

#### 表关系结构

| 表关系                                | 关联字段                                            | 说明                  |
| ---------------------------------- | ----------------------------------------------- | ------------------- |
| products : product\_skus           | products.id → product\_skus.product\_id         | 1:n 关系，一个商品可以有多个SKU |
| activities : activity\_products    | activities.id → activity\_products.activity\_id | 1:n 关系，一个活动可以有多个商品  |
| activity\_products : product\_skus | activity\_products.sku\_id → product\_skus.id   | 1:1 关系，活动商品绑定到具体SKU |

**绝对规则**：`products.stock = 所有 product_skus.stock 的总和`

#### 库存管理关系

| 表名                 | 库存字段  | 作用    | 管理策略                   |
| ------------------ | ----- | ----- | ---------------------- |
| products           | stock | 商品总库存 | 作为商品的总库存展示，等于所有SKU库存之和 |
| product\_skus      | stock | SKU库存 | 实际库存管理单位，订单扣减以此为准      |
| activity\_products | stock | 活动库存  | 从SKU库存中冻结出来的活动专用库存     |

#### 库存更新机制

1. **普通商品库存**：
   - 创建订单时，扣减对应SKU的库存（product\_skus.stock）
   - 订单取消时，恢复对应SKU的库存
   - 每次SKU库存变更时，实时更新products.stock为所有SKU库存之和
2. **活动创建时**：
   - 检查对应SKU的当前库存是否足够
   - 设置 `activity_products.stock` 为活动需要的库存数量
   - 从 `product_skus.stock` 中扣除活动需要的库存数量
   - 更新 `products.stock` 为扣除后的所有SKU库存总和
   - 相当于创建活动后，一定数量库存被冻结给activity专用
   - 后台管理者看到的sku.stock是刨除活动库存后的可用数量
3. **活动订单生成时**：
   - 只扣减 `activity_products.stock`
   - 不影响普通SKU库存
4. **活动结束后可选操作**：

   **第1种情况：剩余活动库存转回普通库存**
   - `product_skus.stock += activity_products.stock`
   - 更新 `products.stock` 为所有SKU库存的新总和
   - 清空 `activity_products.stock`
   **第2种情况：剩余活动库存直接清零**
   - 将 `activity_products.stock` 设为 0
   - 剩余库存不再转回普通库存
5. **库存同步策略**：
   - 实时同步：每次SKU库存变更时立即更新商品总库存
   - 定时同步：通过定时任务确保商品总库存与SKU库存一致
   - 事务保证：所有库存操作必须在事务中执行，确保数据一致性

### 3.1 开发顺序

1. **商品创建逻辑调整** - 基础功能，优先级高（自动创建默认SKU）
2. **商品图片管理功能** - 基础功能，优先级高
3. **商品详情页功能** - 基础功能，优先级高
4. **商品SKU管理功能** - 核心功能，优先级高
5. **库存管理关系实现** - 依赖于SKU功能，优先级中

### 3.2 技术栈

- **后端**：Golang、Gin框架
- **前端**：Vue.js、Element UI
- **富文本编辑器**：Quill.js或TinyMCE
- **数据库**：MySQL

### 3.3 开发注意事项

1. **软删除**：所有删除操作均使用status字段进行软删除
2. **数据一致性**：确保库存数据的一致性和准确性
3. **性能优化**：图片上传和富文本内容存储的性能优化
4. **安全性**：富文本内容的安全过滤，防止XSS攻击
5. **用户体验**：提供直观的操作界面，特别是SKU属性配置

   - 提供直观的SKU属性配置界面，减少操作复杂度
